## Overview

PlayOS is a mobile-first web app where **hosts** (pitch owners) create football games and **players** discover, book, and pay for spots. Think "Eventbrite for pickup football." The platform collects a service fee from players and a listing fee from hosts on every booking.

Use **Replit's built-in PostgreSQL database**, **Replit Auth** (or custom auth as described below), and **Stripe** for payments. Build with **Next.js (App Router)**, **React**, and **Tailwind CSS**. The app must be fully responsive (mobile-first) and support **English** and **Arabic (RTL)**.

## Fee Structure

- **Player Service Fee**: SAR 2 added on top of the game price at checkout
- **Host Listing Fee**: SAR 1 deducted per paid booking from the host's payout
- Example: Game price SAR 50, 10 paid players → Player pays SAR 52, Host receives (50 - 1) × 10 = SAR 490

## User Roles & Auth

### Players
- Sign up with: email, password, phone (Saudi format: 05XXXXXXXX), name
- Sign in with email + password
- Phone stored as +966XXXXXXXXX (strip leading 0, prepend +966)
- Session-based auth with secure HTTPOnly cookies (30-day expiry)
- Can browse games and view game details without account (guest browsing)
- Must authenticate to book/pay

### Organisers (Hosts)
- Apply at `/host/login` → Submit name, pitch name, phone, city → stored in `host_applications`
- After manual approval: account created with phone + password
- Login with phone + password (password hashed SHA256 with salt `playos_2025`)
- Redirected to `/dashboard` after login

## Payment Integration (Stripe)

Use **Stripe Checkout** for payments.

### Booking & Payment Flow
1. Player selects a spot on the pitch layout
2. If not authenticated, redirect to `/auth?returnUrl=/game/[id]`
3. Player clicks "Book Spot" → Server creates Stripe Checkout Session:
   - Amount: game price + SAR 2 service fee
   - Currency: SAR
   - Metadata: game_id, user_id, team, slot_index
   - Success URL: `/payment/callback?session_id={CHECKOUT_SESSION_ID}&gameId=[id]&userId=[uid]&team=[t]&slot=[s]`
   - Cancel URL: `/game/[id]`
4. Player completes payment on Stripe
5. **Webhook** (`/api/payment/webhook`): On `checkout.session.completed`:
   - Verify session metadata
   - Create booking with payment_status = 'paid'
   - If requested slot is taken, auto-assign next free slot on same team
   - If game reaches capacity, update status to 'full'
6. **Callback page** (`/payment/callback`): Verify payment, show confirmation, redirect to game page

### Slot Assignment Logic
- Prefer the player's requested team + slot
- If that slot was taken during payment, find next free slot on same team
- If team is full, find slot on other team
- Only `paid` bookings count as filled slots

## Auto-Cancellation Logic

When fetching open games (in `getAllOpenGames` or `getGamesByOrganiser`):
- Check each game: if `kickoff_time - auto_cancel_hours` has passed AND game is not at capacity
- Automatically update status to `cancelled`
- This runs passively on every game list fetch (no cron needed)

## Refund Policy

Display on `/policies/refund`:

**Player Cancellations:**
- More than 12 hours before kickoff → 100% refund
- 6-12 hours before kickoff → 50% refund
- Less than 6 hours before kickoff → No refund

**Game Cancellations by Host:**
- More than 12 hours before → 100% refund to all players
- 12 hours or less before → 100% refund + host pays SAR 20 late cancellation penalty

## Security & Validation

- **Rate limiting** on all auth endpoints and payment initialization:
  - 20 requests per 60s globally per IP
  - 10 requests per 60s per identifier (email/phone)
  - Return 429 with Retry-After header
- **Input validation**:
  - Email: RFC 5322 pattern, max 254 chars
  - Password: 4-72 chars
  - Phone: Saudi format validation
  - Game title: 2-80 chars, sanitize control characters
  - Capacity: 6-22, must be even
  - Price: positive number
- **Protected routes**: Middleware checks auth cookie, redirects to `/auth` with `returnUrl`
- **Organiser routes**: Additional check that user role = 'organiser'
- **Payment webhook**: Verify Stripe signature before processing

## Key Business Rules

1. **Even capacity only** — Games must have even number of max players (for balanced teams)
2. **One booking per player per game** — Prevent duplicate bookings
3. **Paid = confirmed** — Only paid bookings hold slots. Pending bookings don't count toward capacity.
4. **Auto-balance teams** — When no specific slot selected, assign to team with fewer players
5. **Public vs Private games** — Public games appear in `/games` browse list. Private games are link-only.
6. **Phone normalization** — All Saudi phones stored as +966XXXXXXXXX
7. **Organiser can only manage their own games** — Enforce in all queries
